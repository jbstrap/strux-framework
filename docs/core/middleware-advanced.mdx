# Advanced Middleware Patterns

This document builds on the core Middleware concepts and demonstrates **real-world patterns** you will use in production applications.

---

## Before vs After Middleware

Every PSR-15 middleware has **two execution phases**:

1. **Before** the controller runs  
2. **After** the controller returns a response  

This makes middleware ideal for **cross-cutting concerns**.

---

## Example: Timing Middleware

```php
namespace App\Http\Middleware;

use Psr\Http\Message\ResponseInterface;
use Psr\Http\Message\ServerRequestInterface;
use Psr\Http\Server\MiddlewareInterface;
use Psr\Http\Server\RequestHandlerInterface;

class RequestTimingMiddleware implements MiddlewareInterface
{
    public function process(
        ServerRequestInterface $request,
        RequestHandlerInterface $handler
    ): ResponseInterface {
        // BEFORE controller
        $start = microtime(true);

        $response = $handler->handle($request);

        // AFTER controller
        $duration = microtime(true) - $start;

        return $response->withHeader(
            'X-Response-Time',
            number_format($duration * 1000, 2) . 'ms'
        );
    }
}
````

### Typical Use Cases

**Before**

* Authentication
* Authorization
* Input validation
* Request transformation

**After**

* Logging
* Metrics
* Caching headers
* Compression
* CORS headers

## Error-Handling Middleware

Error-handling middleware wraps the **entire request pipeline** and catches unhandled exceptions.

This should be registered as **global middleware** and as early as possible.

---

### Example: Exception Handling Middleware

```php
namespace App\Http\Middleware;

use Throwable;
use Psr\Http\Message\ResponseInterface;
use Psr\Http\Message\ServerRequestInterface;
use Psr\Http\Server\MiddlewareInterface;
use Psr\Http\Server\RequestHandlerInterface;
use Strux\Component\Http\Response;

class ErrorHandlingMiddleware implements MiddlewareInterface
{
    public function process(
        ServerRequestInterface $request,
        RequestHandlerInterface $handler
    ): ResponseInterface {
        try {
            return $handler->handle($request);
        } catch (Throwable $e) {
            // Log exception here if needed

            return new Response(
                500,
                ['Content-Type' => 'application/json'],
                json_encode([
                    'error' => 'Internal Server Error',
                    'message' => $e->getMessage(),
                ])
            );
        }
    }
}
```

## Authentication Middleware

Authentication middleware verifies **who the user is**.

It typically:

* Reads headers / cookies / tokens
* Resolves the user
* Attaches the user to the request

---

### Example: Auth Middleware

```php
namespace App\Http\Middleware;

use Psr\Http\Message\ResponseInterface;
use Psr\Http\Message\ServerRequestInterface;
use Psr\Http\Server\MiddlewareInterface;
use Psr\Http\Server\RequestHandlerInterface;
use Strux\Component\Http\Response;
use App\Services\AuthService;

class AuthenticateUser implements MiddlewareInterface
{
    public function __construct(
        protected AuthService $auth
    ) {}

    public function process(
        ServerRequestInterface $request,
        RequestHandlerInterface $handler
    ): ResponseInterface {
        $user = $this->auth->userFromRequest($request);

        if (!$user) {
            return new Response(401, [], 'Unauthenticated');
        }

        // Attach authenticated user to request
        $request = $request->withAttribute('user', $user);

        return $handler->handle($request);
    }
}
```

## Role-Based Authorization Middleware

Authorization answers the question:

> “Is this user allowed to do this?”

This is usually **route-specific**.

---

### Example: Role Middleware (Reusable)

```php
namespace App\Http\Middleware;

use Psr\Http\Message\ResponseInterface;
use Psr\Http\Message\ServerRequestInterface;
use Psr\Http\Server\MiddlewareInterface;
use Psr\Http\Server\RequestHandlerInterface;
use Strux\Component\Http\Response;

class RequireRole implements MiddlewareInterface
{
    public function __construct(
        protected string $role
    ) {}

    public function process(
        ServerRequestInterface $request,
        RequestHandlerInterface $handler
    ): ResponseInterface {
        $user = $request->getAttribute('user');

        if (!$user || $user->role !== $this->role) {
            return new Response(403, [], 'Forbidden');
        }

        return $handler->handle($request);
    }
}
```

---

### Using Role Middleware on Routes

```php
use Strux\Component\Attributes\Route;
use Strux\Component\Attributes\Middleware;
use App\Http\Middleware\AuthenticateUser;
use App\Http\Middleware\RequireRole;

class AdminController
{
    #[Route('/admin')]
    #[Middleware(AuthenticateUser::class)]
    #[Middleware(new RequireRole('admin'))]
    public function dashboard()
    {
        // Admin-only access
    }
}
```

## Middleware Order Best Practices

### Global Stack Order

```text
ErrorHandlingMiddleware
→ SessionMiddleware
→ AuthenticationMiddleware
→ Routing
```

### Route-Level Order

```text
Authentication
→ Authorization
→ Controller
```

---

## Summary

* Middleware runs **before and after** controllers
* Error-handling middleware should wrap the entire pipeline
* Authentication identifies the user
* Authorization enforces permissions
* Request attributes are the preferred data-passing mechanism
* PSR-15 enables ecosystem-wide compatibility
